Purpose

Goal

Why this system exists, what problem it solves, and what it does.

The Problem

An organisation has thousands of internal PDFs stored on SharePoint. Staff need answers from these documents quickly but face four hard blockers:

📂
Too many documents
Thousands of PDFs — manually searching is impractical and time-consuming.
🖼️
Scanned images
Many PDFs are scanned paper documents with no machine-readable text — standard search fails entirely.
🔒
Sensitive data
Documents are confidential and cannot be sent to OpenAI, Anthropic, or any external cloud service.
🏢
On-premise only
All processing must happen on the organisation's own hardware — a Mac Mini.

What This System Does

Knowledge Companion is a private, conversational interface over the entire document library — powered entirely by local Ollama models.

📥
Ingests documents
Reads PDFs from a local folder. Extracts text from native PDFs and runs OCR on scanned images automatically.
🧠
Understands content
Uses hierarchical chunking and local embeddings (nomic-embed-text) to enable meaning-based search, not keyword matching.
🤖
Routes intelligently
llama3.2:3b classifies query complexity and routes to the right model — fast answers stay fast, complex reasoning gets the full 8b model.
💬
Answers questions
User asks in plain English. System finds relevant chunks via hybrid search, re-ranks them, and generates a grounded answer.
📎
Cites sources
Every answer includes source document, page number, and a confidence score so the user can verify directly.
🗂️
Multi-tab sessions
ChatGPT-style UI with multiple simultaneous chat sessions. Context switches intelligently when changing tabs.
📁
Folder scoping
Each chat session can be scoped to specific folders — query only HR docs, or only safety manuals, or everything.
🛡️
No hallucinations
Score threshold, strict system prompt, temperature=0, multi-chunk agreement, and source citations layer together to prevent fabrication.

Fully local RAG system. React frontend talks to a FastAPI backend. All intelligence runs through Ollama — three local models handle embeddings, routing, re-ranking, and generation.

🔒
100% Private — React, FastAPI, Ollama, PostgreSQL+pgvector, and OCR all run on-device. Zero external API calls at runtime.

Ollama Only

Ollama Model Strategy

Three models. One runtime. Zero external dependencies. Every intelligent task in the pipeline is handled by an Ollama model.

Model Lineup

Pull once, use forever. All models run on Apple Metal via Ollama.

nomic-embed-text
Embedder — always loaded
  • Convert document chunks to 768-dim vectors at ingestion
  • Convert user queries and HyDE hypothetical answers to vectors
  • Powers pgvector HNSW similarity search
~300 MB always warm 768-dim output
llama3.2:3b
Fast Worker — handles 80% of tasks
  • HyDE: generate a hypothetical document excerpt to improve retrieval (70% HyDE + 30% original blend)
  • Re-ranker: score each retrieved chunk for relevance, keep top 5
  • Router: classify query as SIMPLE or COMPLEX to pick the right answer model
  • Conflict detector: check if top 2 retrieved chunks contradict each other
  • Topic-switch detector: classify follow-up vs new topic for history trimming
  • Answer generator: handles all SIMPLE queries end-to-end
~2 GB ~200-500ms per task temperature=0
llama3.1:8b
Reasoner — called only for complex queries
  • Multi-document comparison and synthesis
  • Complex reasoning across long context windows
  • Queries requiring step-by-step chain of thought
~5 GB ~5-8s response temperature=0

Performance

Compute Pipeline

Five gates filter each query before it reaches the LLM. Most queries never reach the expensive model — saving seconds of latency per response.

Compute Gates

Each gate is a cheap check that can short-circuit the rest of the pipeline.

1

Input Validation

Gibberish detection (vowel-ratio check) rejects keyboard mash. Prompt injection blocking (pattern match) blocks "ignore previous instructions", "pretend to be", and similar attacks. Both checks run before any LLM call.

~0ms — pure Python, no LLM
2

HyDE Expansion (llama3.2:3b)

Generate a hypothetical document excerpt that would answer the question. Blend 70% HyDE + 30% original question as the search query — produces far better vector matches against real document chunks.

~200ms — llama3.2:3b
3

Three-Lane Retrieval + Re-ranking (llama3.2:3b)

Lane 1: pgvector semantic search top 20. Lane 2: BM25 re-order over semantic candidates. Lane 3: PostgreSQL ILIKE exact keyword search for up to 2 extracted acronyms/terms. Merge all lanes via Reciprocal Rank Fusion (RRF), then llama3.2:3b scores each chunk and keeps top 5. Keyword-boost guarantee: if no keyword-matched chunk lands in top 5, the best keyword chunk above threshold is swapped in. Source diversity cap: max 2 chunks per source document (keyword-boosted chunks exempt). If best score is below 0.45 threshold — stop here, return no-docs response. No LLM call needed.

miss: return fallback — no LLM call
4

Model Routing (llama3.2:3b)

Classify the query as SIMPLE (single fact, direct answer) or COMPLEX (comparison, multi-doc, reasoning). Route accordingly.

SIMPLE: llama3.2:3b — ~1-2s total
5

Answer Generation (llama3.2:3b or llama3.1:8b)

Generate answer with temperature=0, streaming enabled. Confidence score (HIGH ≥0.70, MEDIUM ≥0.45) is derived from retrieval re-rank scores before generation — no separate LLM call required, enabling true token streaming from the first token.

COMPLEX: llama3.1:8b — ~5-8s total

Accuracy

Anti-Hallucination

Six independent layers. Each one alone reduces hallucinations — together they make fabrication extremely unlikely.

Layer 1 — Score Threshold Gate

If no retrieved chunk scores above 0.45 similarity, the LLM is never called. The system returns a fixed fallback message. No generation means no hallucination.

graph LR A[Query] --> B[Retrieve and Re-rank] B --> C{Best score above 0.45} C -->|No| D[Fallback: no relevant documents found] C -->|Yes| E[Proceed to LLM] style D fill:#222535,stroke:#fb923c,color:#fb923c style E fill:#222535,stroke:#34d399,color:#34d399

Layer 2 — Strict System Prompt

The LLM is instructed to answer only from the provided context and to explicitly admit when it cannot find the answer.

graph TD A[System Prompt sent with every request] --> B[Answer ONLY using the context below] A --> C[If answer not in context say so explicitly] A --> D[Do not use any outside knowledge] A --> E[Think step by step before answering] style A fill:#222535,stroke:#6c8ef7,color:#6c8ef7

Layer 3 — Temperature Zero

Both models run at temperature=0. This makes the model deterministic — it always picks the highest-probability token, eliminating creative deviation from the source material.

SettingEffect
temperature=0Deterministic output — no randomness or creative generation
top_p=1No nucleus sampling — sticks to most probable tokens
repeat_penalty=1.1Discourages repetitive filler content

Layer 4 — Multi-Chunk Agreement

llama3.2:3b checks if retrieved chunks agree before passing them to the answer model. Conflicts are surfaced to the user rather than silently resolved.

Layer 5 — Source Citations + Pre-Generation Confidence Score

Every answer surfaces the source document and page number. The user can open the PDF and verify the exact passage. If the citation doesn't match — hallucination is caught immediately.

📄
Filename
Exact PDF name shown under each answer
🔢
Page number
Specific page within the document
📊
Confidence score
HIGH ≥0.70 / MEDIUM ≥0.45 / LOW — derived from retrieval re-rank scores before generation, enabling true token streaming
🔗

How many chunks agreed — 2+ agreement = higher confidence

Layer 6 — Input Validation

Two lightweight checks fire before any LLM work — stopping wasted compute and preventing prompt injection at the boundary.

🚫
Prompt Injection Blocking
Pattern match blocks "ignore previous instructions", "pretend to be", and similar injection attempts before any LLM call

Multi-Tab Architecture

How Context Switching Works

Every API request carries the session_id. FastAPI loads the correct history from PostgreSQL. The LLM never sees context from a different tab.

Context Window Management

Sliding window of last 12 turns sent to the LLM. Adaptive topic-switch detection (llama3.2:3b) checks whether the new question is a follow-up or a new topic — on topic switch, history trims to 4 turns to avoid polluting the new context. Older turns stay in PostgreSQL for history display only.
